## 一、一句话理解整个系统群

天枢银行核心系统群做的事情，本质上就是：**管钱、算钱、记钱、防风险**。

- **管钱**：客户的钱存在哪（账务系统）、通过什么方式流动（支付系统）、卡和账户怎么关联（卡系统）
- **算钱**：贷款利息怎么算（信贷核心）、手续费怎么分（清结算）、产品参数怎么定（产品中心）
- **记钱**：每一笔交易都有据可查（交易系统）、每一分钱的流向都能对上（对账系统）
- **防风险**：交易前拦截可疑操作（风控系统）、客户身份真实可信（客户中心 + 用户中心）、额度不超限（额度中心）

## 二、系统分组与角色

### 资金流转主线（钱怎么动）

```
渠道/用户
   │
   ▼
┌──────────┐    支付指令    ┌──────────┐    记账请求    ┌──────────┐
│ 交易系统  │ ──────────▶  │ 支付系统  │ ──────────▶  │ 账务系统  │
│Transaction│              │ Payment  │              │Accounting│
└──────────┘              └──────────┘              └──────────┘
                               │                        │
                               ▼                        ▼
                          外部渠道                   账户余额变动
                       (银联/网联/跨境)              (借贷平衡)
```

这是最核心的一条线：**交易发起 → 支付执行 → 账务记账**。几乎所有即时金融业务都走这条路。

### 信贷业务线（借钱还钱）

```
客户申请贷款
   │
   ▼
┌──────────────┐   查产品参数   ┌──────────┐
│ 信贷流程管理  │ ◀──────────  │ 产品中心  │
│  LoanProcess │              │ProductCtr│
└──────────────┘              └──────────┘
   │  编排流程
   ▼
┌──────────────┐   放款/还款   ┌──────────┐   记账   ┌──────────┐
│  信贷核心    │ ──────────▶  │ 支付系统  │ ──────▶ │ 账务系统  │
│  LoanCore   │              │ Payment  │         │Accounting│
└──────────────┘              └──────────┘         └──────────┘
   │
   ▼
┌──────────┐
│ 额度中心  │  ← 放款占用额度，还款释放额度
│CreditLimit│
└──────────┘
```

信贷流程管理是"指挥官"（编排进件→审批→签约→放款→贷后），信贷核心是"计算引擎"（算利息、生成还款计划）。两者是 Partnership 关系，协同演进。

### 事后保障线（对不对得上）

```
┌──────────┐  交易流水
│ 交易系统  │ ─────────┐
└──────────┘          │
┌──────────┐  支付流水  │    ┌──────────────┐
│ 支付系统  │ ─────────┼──▶│   对账系统    │ → 差异处理
└──────────┘          │    │Reconciliation│
┌──────────┐  账务流水  │    └──────────────┘
│ 账务系统  │ ─────────┘
└──────────┘

交易完成后：
┌──────────┐  交易完成事件  ┌──────────┐  结算指令  ┌──────────┐
│ 交易系统  │ ───────────▶ │ 清结算系统 │ ────────▶ │ 支付系统  │
└──────────┘              │ Clearing  │          └──────────┘
                          └──────────┘
```

### 基础支撑（谁在用、谁能用、用什么产品）

```
┌──────────┐  技术身份   ┌──────────┐  业务身份   ┌──────────┐
│ 用户中心  │ ────────▶  │ 客户中心  │ ────────▶  │ 卡系统    │
│UserCenter│  (登录认证) │CustomerCtr│  (KYC/画像) │CardSystem│

└──────────┘
```

| 用户中心 | 登录认证、权限管理 | 通用 | 门禁系统 |

| 产品中心 | 金融产品定义与参数配置 | 核心 | 产品货架 |

## 四、典型业务场景端到端走查

### 场景 1：一笔银行转账（张三转 1000 元给李四）

```
1. 张三在 App 发起转账
       │
2. 交易系统 受理交易，生成交易流水，做幂等校验
       │
3. 交易系统 → 风控系统：实时风控评估
       │                    ├─ 通过 → 继续
       │                    └─ 拒绝 → 交易终止
       │
4. 交易系统 → 额度中心：校验可用额度（如有信用额度场景）
       │
5. 交易系统 → 支付系统：发起支付请求
       │
6. 支付系统 判断路由：
       │   ├─ 同行转账 → 直接驱动账务系统
       │   └─ 跨行转账 → 走银联/网联渠道
       │
7. 支付系统 → 账务系统：记账请求
       │   ├─ 借：张三账户 1000 元
       │   └─ 贷：李四账户 1000 元（或在途户）
       │
8. 账务系统 完成复式记账，更新余额
       │
9. 支付系统 → 交易系统：支付完成事件
       │
10. 交易系统 更新交易状态为"成功"
       │
11. 事后：
    ├─ 清结算系统 订阅交易完成事件，清分手续费
    ├─ 对账系统 定期拉取三方流水核对
    └─ 张三收到转账成功通知
```

**涉及系统**：交易 → 风控 → 额度(可选) → 支付 → 账务 → 清结算 → 对账

---

### 场景 2：一笔贷款放款（李四申请消费贷 5 万元）

```
1. 李四在 App 提交贷款申请
       │
2. 信贷流程管理 受理进件
       │
3. 信贷流程管理 → 客户中心：查询客户信息和 KYC 状态
       │                         └─ KYC 未通过 → 流程终止
       │
4. 信贷流程管理 → 产品中心：查询产品参数（利率、期限、还款方式）
       │
5. 信贷流程管理 → 风控系统：信贷风控评估（信用评分）
       │                       ├─ 通过 → 继续
       │                       ├─ 人工审核 → 转审批岗
       │                       └─ 拒绝 → 流程终止
       │
6. 信贷流程管理 → 额度中心：预占用额度 5 万
       │                       └─ 额度不足 → 流程终止
       │
7. 审批通过，签约完成
       │
8. 信贷流程管理 → 信贷核心：生成还款计划
       │   └─ 信贷核心 → 产品中心：获取利率等计算参数
       │   └─ 生成 12 期还款计划（等额本息）
       │
9. 信贷流程管理 → 信贷核心：执行放款
       │
10. 信贷核心 → 额度中心：正式占用额度（预占用 → 正式占用）
       │
11. 信贷核心 → 支付系统：放款支付请求（5 万元划入李四账户）
       │
12. 支付系统 → 账务系统：记账
       │   ├─ 借：贷款发放户 5 万
       │   └─ 贷：李四活期账户 5 万
       │
13. 放款成功，李四收到 5 万元
```

**涉及系统**：信贷流程 → 客户中心 → 产品中心 → 风控 → 额度中心 → 信贷核心 → 支付 → 账务

---

### 场景 3：一笔贷款还款（李四还第 1 期月供 4500 元）

```
1. 到期日，信贷核心发起自动扣款
       │
2. 信贷核心 → 支付系统：还款代扣请求 4500 元
       │
3. 支付系统 → 账务系统：记账
       │   ├─ 借：李四活期账户 4500 元
       │   └─ 贷：贷款本金户 + 利息收入户（按还款计划拆分）
       │
4. 支付系统 → 信贷核心：支付成功事件
       │
5. 信贷核心 更新还款计划状态（第 1 期 → 已还）
       │
6. 信贷核心 → 额度中心：释放已还本金对应的额度
       │   └─ 循环额度场景下，可用额度恢复
       │
7. 信贷核心 → 信贷流程管理：还款完成事件
       │
8. 李四收到还款成功通知
```

**涉及系统**：信贷核心 → 支付 → 账务 → 额度中心 → 信贷流程

## 五、常见疑问

### Q1：交易系统和支付系统为什么要分开？

交易系统管的是"业务语义"（这是一笔转账还是消费），支付系统管的是"资金执行"（走哪个渠道、怎么拆单）。一笔交易可能对应多笔支付（比如拆单），一笔支付也可能被多个业务场景复用。

### Q2：信贷流程管理和信贷核心为什么要分开？

信贷流程管理是"指挥官"，负责编排业务流程（进件→审批→签约→放款→贷后）；信贷核心是"计算引擎"，负责纯粹的信贷计算（利息、还款计划、罚息）。流程可能因业务需求频繁变化，但计算逻辑相对稳定，分开后各自独立演进。

### Q3：用户中心和客户中心为什么要分开？

用户中心管的是"技术身份"（登录、认证、权限），客户中心管的是"业务身份"（个人客户、企业客户、KYC）。一个用户可以对应多个客户角色（比如既是个人客户又是企业法人代表）。

### Q4：额度中心为什么独立出来？

额度是跨业务的共享资源 — 信贷要用、交易要查、风控要参考。如果把额度逻辑嵌入信贷系统，其他系统就很难复用。独立出来后，额度中心成为额度的"单一事实来源"。

### Q5：清结算和对账有什么区别？

清结算是"算钱"：交易完成后，把手续费、分润等拆清楚，轧差后结算。对账是"查错"：把交易、支付、账务三方的流水拉出来比对，找出不一致的地方。

## 六、上下游关系速记

本项目采用"模型影响力方向"定义上下游：

- **上游（Upstream）**= 模型的权威方 / 服务提供方，模型变更会影响下游
- **下游（Downstream）**= 模型的消费方 / 适配方，需要适配上游模型
- 注意：**调用方向可能与上下游方向相反**（下游调用上游的服务）

最常见的上游系统：
- **账务系统** — 几乎所有涉及资金的系统都依赖它记账（被 5 个系统依赖）
- **产品中心** — 信贷、额度等业务系统都从它获取参数（被 3 个系统依赖）
- **风控系统** — 交易和信贷的关键拦截点（被 2 个系统依赖）
