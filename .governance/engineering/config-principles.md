# 配置设计原则

## P01：变更驱动方决定归属层

一条配置属于哪个上下文/层，取决于它的变更驱动方，而不是它存在于哪张表。

| 驱动方 | 归属层 | 示例 |
|--------|--------|------|
| 产品/业务人员 | 业务配置层 | 利率、期数、费用规则、准入条件 |
| 技术团队（流程编排） | 技术配置层 | 步骤序列、重试次数、流程开关 |
| 技术团队（稳定性） | 技术配置层 | 超时时间、熔断阈值、查询间隔 |
| 运维/部署 | 环境变量/配置中心 | 服务地址、TLS 证书、环境标识 |
| 中间件 | 基础设施配置 | MQ topic、Redis key 前缀 |

**关键规则**：与业务类型绑定的技术参数进领域配置；与部署环境绑定的参数不进领域模型。

## P02：不可变配置

配置记录一旦创建不可修改，变更以新增版本号处理。

- 历史单据天然关联创建时的配置版本
- 配置变更审计链完整
- 简化受理逻辑，不需要在受理时复制配置字段

## P03：业务层与技术层物理隔离

业务配置（产品要素）与技术配置（编排参数）分表管理，维护权限分离，独立版本管理。

## P04：配置存参数，逻辑在代码

配置表存储参数值和开关，业务规则的执行逻辑在代码中。

```
✅ max_retry_count = 3
✅ admission_enabled = true
✅ age_min = 22

❌ admission_rule = "age >= 22 AND region NOT IN ('650000')"
```

规则表达式配置化会导致测试困难、版本管理混乱、调试成本高。

## P05：配置与代码解耦

新增一个产品/业务类型、调整参数，不应该需要代码变更或服务重启。

- 新增产品需要改代码 → 说明有业务逻辑硬编码，需要配置化
- 新增步骤类型需要改代码 → 正常，步骤类型是代码扩展点

## P06：内容不可变，状态可流转

配置内容不可变（只新增不更新），但运营状态（ENABLED/DISABLED）和有效期可以独立变更。

## P07：技术配置可 fallback，业务配置必须显式

- 技术配置可以有系统级默认值，未配置时 fallback 到默认
- 业务配置必须显式配置，不允许 fallback（如利率为 NULL 时直接拒绝，不猜测）
