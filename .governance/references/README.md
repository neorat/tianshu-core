# 参考文献模块

> 外部参考资料的索引和摘要，供 AI 在做决策时引用。

## 设计理念

项目开发过程中会参考大量外部资料——官方文档、技术文章、监管文件、其他 AI 的分析结论。
这些资料分散在各处，难以复用。

本模块将这些参考资料结构化整理，让 AI 在生成方案时能说"根据《xxx》的分析，我们选择方案 A"，
而非仅依赖通用训练数据。

## 目录结构

```
references/
├── README.md              # 本文件
├── business/              # 业务领域参考资料
│   └── {domain}/          # 按领域分子目录
│       └── {topic}.md
└── technical/             # 技术参考资料
    └── {topic}/           # 按技术主题分子目录
        └── {article}.md
```

## 文档格式

每篇参考文档遵循以下结构：

```markdown
# 文档标题

## 来源
- 原文链接：https://...
- 作者 / 机构：xxx
- 日期：YYYY-MM-DD

## 摘要
（核心观点，AI 可直接引用）

## 与本项目的关联
（为什么这篇文档对我们有用）

## 关键结论
- 结论 1
- 结论 2
```

## 与其他模块的区别

| 模块 | 内容 | 示例 |
|------|------|------|
| `references/` | 别人说了什么 | 央行支付清算规则摘要 |
| `architecture/decisions/` | 我们决定了什么 | ADR-001: 采用 Saga 做分布式事务 |
| `domain/` | 我们的模型是什么 | 聚合定义、统一语言 |
| `engineering/` | 我们怎么写代码 | 编码规范、异常处理 |

## 使用场景

- 做需求分析时，引用业务领域资料
- 做技术选型时，引用方案对比文章
- 做合规检查时，引用监管文件要点
- 复用之前问过其他 AI 的结论

## AI 使用方式

在 brainstorming 工作流中，AI 应该：

1. **检查 references** — 根据需求主题，查看 `references/business/` 或 `references/technical/` 下是否有相关文档
2. **引用已有资料** — 如果有，在提出方案时引用："根据《xxx》的分析..."
3. **补充缺失资料** — 如果没有或不够，可用 web search 补充，或向用户询问
4. **建议新增** — 如果发现有价值的外部资料（用户提供的链接、其他 AI 的结论、查到的文章），主动建议用户添加到 references

## 命名规范

- 目录名：小写 kebab-case，按领域 / 技术主题划分
- 文件名：小写 kebab-case，能体现资料主题
