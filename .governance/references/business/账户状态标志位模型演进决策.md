# 账户状态标志位模型演进决策

## 来源
- 链接：Kiro brainstorming 对话
- 来源：Kiro + 用户协作
- 时间：2026-04-23

## 问题

原"账户状态机设计决策"采用 restriction_status 单枚举（NONE / PARTIAL_FREEZE / FULL_FREEZE / STOP_PAYMENT）表达交易限制。实际业务中，止付与冻结是两个独立维度，可能同时存在（如一个账户同时被风控止付且有司法冻结），单枚举无法表达叠加状态。此外，预授权冻结只冻结金额不限制交易方向，与司法冻结（冻结金额 + 禁止出入账）的行为完全不同，需要更灵活的模型。

## 结论

**去掉 restriction_status 枚举，改为 lifecycle_status（主状态枚举）+ 独立标志位 + 冻结明细驱动的模型。**

### 账户状态完整模型

| 维度 | 字段 | 类型 | 值 / 说明 |
|------|------|------|-----------|
| 生命周期 | lifecycle_status | 枚举 | ACTIVE / PENDING_CLOSE / CLOSED |
| 出账控制 | debit_blocked | boolean | true = 禁止出账 |
| 入账控制 | credit_blocked | boolean | true = 禁止入账 |
| 运营治理 | activity_status | 枚举 | NORMAL / DORMANT / UNCLAIMED |
| 冻结金额 | frozen_amount | BigDecimal | 派生自冻结明细合计，影响可用余额 |

### 交易控制组合效果

| debit_blocked | credit_blocked | 效果 | 典型场景 |
|---------------|----------------|------|---------|
| false | false | 正常交易 | 默认状态 |
| true | false | 禁止出账，允许入账 | 风控止付、反诈止付 |
| false | true | 允许出账，禁止入账 | 极少见，特殊监管 |
| true | true | 禁止出入账 | 司法冻结 |

### 运营治理状态迁移

```text
NORMAL → DORMANT → NORMAL（客户激活）
DORMANT → UNCLAIMED
UNCLAIMED → NORMAL（特殊领回流程）
```

### 各字段触发条件与影响

| 字段 | 触发条件 | 对交易的影响 |
|------|---------|-------------|
| lifecycle_status = ACTIVE | 开户完成 | 允许交易的前提 |
| lifecycle_status = PENDING_CLOSE | 发起销户 | 禁止新交易，等待清理 |
| lifecycle_status = CLOSED | 销户完成 | 终态，仅可查询历史 |
| debit_blocked = true | 风控止付 / 司法冻结 / 反诈止付 | 禁止出账 |
| credit_blocked = true | 司法冻结 / 特殊监管 | 禁止入账 |
| activity_status = DORMANT | 系统自动（12个月无交易）或人工标记 | 需激活后才能交易 |
| activity_status = UNCLAIMED | 系统自动（久悬认定）或人工标记 | 特殊监管，需领回流程 |
| frozen_amount > 0 | freeze() 操作 | 可用余额减少 |

### 记账准入判断

```
lifecycle == ACTIVE
&& activity_status == NORMAL
&& (direction == OUT ? !debit_blocked && available_balance >= amount : !credit_blocked)
```

## 关键论据

### 1. 冻结与止付彻底解耦

冻结只管金额（frozen_amount），交易方向限制用独立标志位（debit_blocked / credit_blocked）。两者可以独立设置、独立解除，互不干扰。

典型场景对比：

| 场景 | 冻结操作 | 标志位操作 |
|------|---------|-----------|
| 预授权冻结 | freeze(amount, PRE_AUTH) | 不设标志位，账户仍可正常出入账 |
| 风控止付 | 不冻结金额 | debit_blocked = true |
| 司法冻结 | freeze(amount, JUDICIAL) | debit_blocked = true + credit_blocked = true |
| 反诈止付 | 不冻结金额 | debit_blocked = true |

### 2. 预授权场景的完整支持

冻结类型扩展为 JUDICIAL / RISK / PLEDGE / PRE_AUTH。预授权完成时提供原子操作 `debitWithUnfreeze(freezeId, amount)`，避免"解冻后未扣款"的中间状态被其他交易抢占可用余额。

### 3. 标志位变更需要审计

所有标志位和状态变更记录在通用审计日志表中：

| 字段 | 说明 |
|------|------|
| audit_id | 审计记录 ID |
| account_id | 账户 ID |
| change_type | 变更类型（BLOCK_DEBIT / UNBLOCK_DEBIT / BLOCK_CREDIT / UNBLOCK_CREDIT / MARK_DORMANT / MARK_UNCLAIMED / REACTIVATE / LIFECYCLE_CHANGE 等） |
| old_value | 变更前值 |
| new_value | 变更后值 |
| reason | 变更原因 |
| source_system | 来源系统 |
| operator_id | 操作员 |
| created_at | 变更时间 |

### 4. 并发控制

Account 聚合根采用乐观锁（version 字段），Phase 1 满足 MVP 需求。Phase 2 针对热点账户（手续费收入户、清算过渡户等）引入异步合并记账优化。

## 对本项目的参考价值

此决策是对原"账户状态机设计决策"的演进。原方案的 restriction_status 单枚举在止付+冻结叠加、预授权等场景下表达力不足。新方案用标志位替代枚举，每个标志独立管理，组合灵活，且与冻结明细完全解耦。spec 文档中 FR-8（账户状态机）和聚合设计中 Account 聚合的字段、领域方法、不变量需要同步更新。
