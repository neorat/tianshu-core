# 账务系统科目体系设计决策

## 来源
- 链接：https://chatgpt.com/share/69e650fa-cfa8-8399-a1e6-542e6e2bcf1c
- 时间：2025-04-21

## 问题

银行账务的核心是科目体系，账户需要挂科目才能记账。科目决定了账户的性质（资产/负债/收入/支出）和记账方向。三种设计模式如何决策：

- A. 账务系统内置科目体系：账务系统自己管理科目树，开户时指定科目代码。适合账务系统作为独立核心的场景。
- B. 科目体系外部化：科目由产品中心或专门的参数管理系统维护，账务系统只存储科目代码引用。适合科目与产品强关联的场景。
- C. 暂不建模科目：MVP 阶段先不引入科目概念，账户只记录余额和流水，科目映射后续再加。适合快速验证核心记账逻辑。

## 结论

**银行核心系统群场景，优先选择 B（科目体系外部化），但账务系统内部保留"科目引用能力 + 基础校验缓存"。**

| 场景 | 推荐 |
|------|------|
| 单一业务线、系统少、快速落地 | A |
| 多产品线（存款/贷款/支付/卡/理财） | B |
| 集团化、多法人、多国家、多账套 | B |
| AI驱动平台化建设、领域解耦 | B |
| 只是验证记账引擎/MVP | C（短期） |

## 为什么选 B

### 1. 科目属于企业级主数据，不完全属于账务系统

科目是"财务规则 + 产品参数 + 组织治理"的交叉领域。在核心系统群架构下，旁边还有产品中心、客户中心、支付系统、授信系统、清结算、对账系统、总账/财务系统、参数中心。

### 2. 产品驱动科目变化频繁

- 活期存款产品A → 负债科目 20101
- VIP存款产品B → 负债科目 20102
- 消费贷产品 → 应收贷款科目
- 信用卡分期产品 → 分期应收科目

这些变化常由产品部门驱动，不应每次改账务系统。

**合理边界：** 产品中心定义产品 → 绑定会计科目模板 → 账务系统执行记账

### 3. 多账套/多法人要求

同一个产品在不同法人（新加坡、香港、中国）可能挂不同会计科目。若放账务系统内置，会导致账务系统越来越像 ERP 参数系统。

### 4. 账务系统应聚焦"记账引擎"

核心职责：分录生成、借贷平衡、余额维护、幂等控制、日终结转、对账追踪、高并发事务一致性。而不是维护复杂科目树。

## 推荐架构（B+）

### 主数据归属：参数中心 / 产品中心维护

- 科目主数据、科目树、科目状态
- 法人账套
- 产品-科目映射
- 事件-科目规则、生效日期

### 账务系统本地存储

- subject_code
- subject_name_snapshot
- normal_balance_direction
- subject_type
- version

**原则：** 主数据外部化，但账务系统保留快照/缓存，兼顾高性能、解耦、参数统一治理、可追溯。

## 分阶段落地建议

### Phase 1（当前最优）

采用 **A 的运行模式 + B 的边界设计**：
- 科目表先放账务系统库里
- 但接口设计成未来可外部接入

```java
SubjectService.getSubject(code)
// 未来替换成
ConfigCenterSubjectService
```

### Phase 2

科目迁移到参数中心。

### Phase 3

支持：多法人科目表、生效版本、产品自动挂科、会计事件模板引擎。

## 对本项目的参考价值

天枢银行核心系统群采用多产品线架构，科目应作为企业级主数据管理。短期可先在账务系统内置科目表，但接口必须按外部化设计，为后续迁移到参数中心做准备。
