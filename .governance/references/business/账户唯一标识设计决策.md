# 账户唯一标识设计决策（三层标识体系）

## 来源
- 链接：对话粘贴
- 来源：ChatGPT
- 时间：2026-04-21

## 问题

银行核心系统中，账户的唯一标识应该怎么设计？只用一个 account_no 是否够用？多法人、多渠道、换卡换号、分库分表等场景下如何保证标识体系的可扩展性？

## 结论

**账户至少要有三层标识，不要只用一个 account_no 打天下：**

| 标识 | 用途 | 是否可变 |
|------|------|---------|
| **account_id** | 系统内部唯一主键 | 不可变 |
| **account_no** | 业务账号（对内对外业务使用） | 尽量不变 |
| **external_ref** | 外部渠道号（卡号/虚拟号/IBAN 等） | 可变，一对多 |

## 关键论据

### 1. 为什么不能只用 account_no

只用 `account_no varchar primary key` 的后患：

| 问题 | 说明 |
|------|------|
| 账号规则变化 | 旧规则 16 位，新规则 20 位，主键改不了 |
| 账号迁移 | 并购/换核心时账号保留，但内部系统已变 |
| 一个账户多个号码 | 同一账户绑定卡号A、卡号B、虚拟账号C、IBAN D |
| 卡号 ≠ 账户号 | 卡只是访问介质，不是账户本体 |

### 2. 三层标识详解

#### account_id（真正唯一标识）

- 全局唯一、无业务含义、永不变更
- 推荐生成方式：Snowflake ID / UUIDv7 / KSUID / 数据中心号+序列
- 用途：数据库主键、分布式关联、幂等引用、内部服务调用
- **不建议**：10001、10002 单库自增 ID，未来分库分表必踩坑

#### account_no（业务账号）

- 银行业务真正使用的账户号，可读性较高
- 常见结构：`法人代码 + 产品代码 + 顺序号 + 校验位`（如 SG01-10-0000001234-7）
- 校验位建议：Luhn / Mod11
- 用途：客服查询、用户展示、对账单、柜面操作、渠道传输

#### external_ref（外部映射号）

- 一个账户可绑定多个外部号码：卡号 PAN、虚拟账号 VA、支付平台子商户号、旧系统账号、IBAN
- 通过 identifier 表实现一对多映射

### 3. 推荐数据模型

account 表：
```text
account_id         bigint PK（Snowflake/UUIDv7）
account_no         varchar unique
account_type
customer_id
legal_entity_id
currency
status
created_at
```

account_identifier 表（多号码体系）：
```text
id
account_id
identifier_type    -- ACCOUNT_NO / CARD_NO / IBAN / VIRTUAL_NO / LEGACY_NO / WALLET_NO
identifier_value
status
effective_from
effective_to
```

### 4. 多法人唯一性

银行核心系统群大概率支持多法人（新加坡、香港、菲律宾等），唯一性建议：

- `(legal_entity_id + account_no)` 联合唯一，或
- `account_no` 全球唯一

### 5. 主子账户标识

- 展示层可用：`ACC10001`（主）→ `ACC10001-01`、`ACC10001-02`（子）
- 内部仍然：account_id 独立主键 + parent_account_id 关联

### 6. 虚拟账户标识

- 虚拟账户号码单独体系：VA2026040001、VA2026040002
- 映射到真实账户，不要把虚拟账号当真实账户主键

### 7. API 设计原则

```text
对外接口：{ "accountNo": "SG0110009988" }
内部服务：{ "accountId": "78451234599871" }
```

外部系统不应直接使用内部 ID，有安全与耦合问题。

### 8. 典型错误

| 错误 | 后果 |
|------|------|
| 账号即主键 | 规则变化时无法修改 |
| 账号业务含义过重（编码含地区/产品/年份） | 未来改不了 |
| 卡号 = 账户号 | 卡是介质不是账户 |
| 外部系统直接使用内部 ID | 安全与耦合风险 |

## 对本项目的参考价值

天枢核心系统的账户标识必须采用三层体系：account_id（Snowflake）做内部主键，account_no 做业务账号，account_identifier 表支持多号码映射。这一设计直接影响分库分表、多法人扩展、换卡换号、系统迁移等核心能力，必须在 Phase 1 就确立。
