# 计息职责归属决策（存款计息 vs 贷款计息的领域边界）

## 来源
- 链接：对话粘贴
- 来源：ChatGPT
- 时间：2026-04-21

## 问题

活期账户计息入账应该由账务系统做，还是由产品系统做？贷款计息又该归谁？划分依据是什么？

## 结论

**不按"账户在哪个系统"划分，而按业务语义 + 利息规则复杂度 + 责任边界划分。**

核心原则：**谁掌握业务资产负债逻辑，谁负责计息计算。账务系统负责资金记账与余额真实性。**

| 计息类型 | 计算方 | 入账方 | 原因 |
|---------|--------|--------|------|
| **活期存款计息** | 账务核心（存款子域） | 账务核心 | 余额驱动型，数据天然在账务系统 |
| **贷款/分期计息** | 信贷核心 | 账务核心 | 合同驱动型，依赖还款计划与合同条款 |

一句行业标准话术：**存款利息是余额型计息，贷款利息是合同型计息，两者不应混为一个引擎。**

## 关键论据

### 1. 活期存款计息为什么在账务系统

活期存款计息规则：按日计提、按季结息、日均余额计息、阶梯利率、账户状态过滤。

这些数据账务系统天然具备：每日余额、冻结金额、可用余额、账户状态。

批量流程：
```text
日终批处理 → 扫描活期账户 → 计算应付利息 → 累计待付利息 → 季末结息入账
```

分录：
```text
借：利息支出
贷：客户活期账户
```

本质：**余额驱动型计息**，规则标准化，账务系统直接可算。

### 2. 贷款计息为什么在信贷核心

贷款利息依赖合同：本金、年利率、期数、还款方式、提前还款违约金、逾期罚息、宽限期等。

这不是简单的余额乘利率，它依赖：摊还计划、合同条款、已还未还状态、罚息规则、展期重组规则。这些属于信贷域，不属于账务域。

流程：
```text
信贷核心日终跑批 → 生成应收利息 → 生成应还账单 → 调用账务入账
```

分录：
```text
借：客户贷款账户
贷：利息收入账户
```

**职责链方向：信贷核心调用账务系统记账（不是账务告知信贷）。**

### 3. 计息分两层（银行级最佳实践）

| 层次 | 职责 | 负责方 |
|------|------|--------|
| Layer 1：业务计息引擎 | 算金额：interest / penalty / fee | 存款域 或 信贷域 |
| Layer 2：账务入账引擎 | 做分录：DR/CR、余额更新、流水落库、幂等控制 | Ledger 账务引擎 |

### 4. 系统边界图

```text
┌─────────────────┐
│   产品中心       │
│ 利率参数/产品规则 │
└───────┬─────────┘
    ┌───┴───────────────┐
    │                   │
┌───▼────────────┐  ┌───▼────────────┐
│ 存款/账务核心   │  │   信贷核心      │
│ 活期定期计息    │  │ 贷款/分期计息   │
└───────┬────────┘  └───────┬────────┘
        │                   │
        └─────────┬─────────┘
                  ▼
          ┌──────────────┐
          │ Ledger账务引擎 │
          │ 借贷分录入账   │
          └──────────────┘
```

### 5. 天枢平台模块划分建议

```text
deposit-core      → 活期利息计算
loan-core         → 贷款利息计算
ledger-core       → 统一分录入账、余额更新、流水沉淀
account-core      → 账户生命周期
product-center    → 利率参数、产品规则
batch-scheduler   → 日终批量调度
```

### 6. 初创阶段合并策略

短期可将计息引擎与账务引擎合并，但代码上必须保留边界：

```text
✅ depositInterestService   → 存款计息
✅ loanInterestService      → 贷款计息
❌ calcInterest(accountId)  → 万能方法，后患无穷
```

## 对本项目的参考价值

天枢核心系统的计息职责必须按业务语义划分：存款计息归账务/存款核心，贷款计息归信贷核心，账务引擎统一负责入账。即使初期只有一个账务系统，代码层面也要保留 depositInterestService 和 loanInterestService 的边界，避免后续拆分困难。
