# 虚拟子账户设计决策

## 来源
- 链接：https://chatgpt.com/share/69e650fa-cfa8-8399-a1e6-542e6e2bcf1c
- 时间：2025-04-21

## 问题

虚拟子账户有三种设计模式，如何决策：
- A. 资金隔离型：子账户余额独立，主账户余额 = 各子账户余额之和
- B. 记账隔离型：子账户只是记账维度，资金实际在主账户
- C. 额度控制型：给子账户设置额度上限

## 核心认知

A/B/C 不是互斥选项，而是三种完全不同的账务模型：
- A = 资金所有权 / 余额隔离模型
- B = 核算维度 / 管理视图模型
- C = 额度控制 / 风险约束模型

**很多系统失败，就是把它们混成一个 sub_account 表。**

## 结论

**账务核心只做 A + B 两类账户模型，C 不应作为账户模型，而应作为额度域模型独立建设。**

- A 属于账务账户体系
- B 属于账务统计体系（虚拟账本）
- C 属于授信/额度中心

## 一句话决策法

| 业务诉求 | 选择 |
|---------|------|
| 钱必须真正隔离、可独立出入账 | A |
| 钱不隔离，只要分摊统计 | B |
| 需要控制可用额度/授信额度 | C（独立额度系统） |
| 既要资金隔离又要额度控制 | A + C |
| 既要门店统计又要真实余额 | A + B |
| 复杂金融产品 | A + B + C |

## 三种模型详解

### A. 资金隔离型（真实子账户）

子账户是真账户，独立余额、独立流水、可冻结、可计息、可状态控制。主账户余额 = sum(child.balance)。

适用：存款分钱包、理财持仓资金池、保证金账户、商户待结算资金池。

优点：账实一致、审计友好、风控清晰、易对账。
缺点：账户数爆炸、日终计息复杂、迁移成本高。

### B. 记账隔离型（虚拟核算维度）

钱只有一份在主账户，子账户只是维度账本。技术上不是账户表，而是 account_sub_ledger 或 posting_line.dimension。

适用：商户按门店统计、平台按业务线核算、成本中心归集。

优点：灵活、不产生海量真实账户。
缺点：维度金额与主余额可能不一致、容易被误当真余额。

### C. 额度控制型（必须独立）

这不是余额，是可使用权限。涉及授信审批、风险策略、占用释放、多产品共享额度、临时调额、风控冻结——属于授信域/风控域。

## 推荐数据模型

### Account 表（真实账户，用于 A）

```
account_id, parent_account_id, account_type(MAIN/SUB_REAL), currency, balance
```

### Ledger Dimension 表（虚拟核算，用于 B）

```
account_id, dimension_type, dimension_code, virtual_balance
```

### Limit 表（额度，用于 C，独立系统）

```
customer_id, limit_type, total_limit, used_limit, available_limit
```

## 正确拆域方式

```
账户中心（Account）  → 负责真实资金账户
账务中心（Ledger）   → 负责记账流水
额度中心（Limit）    → 负责额度
产品中心（Product）  → 负责产品配置
客户中心（CIF）      → 负责客户主体
```

## 分阶段落地

### Phase 1
先做：主账户 + 真实子账户（A）+ 记账维度（B），额度先简单化。

### Phase 2
独立额度中心：总额度、子额度、占用释放、临时调额。

### Phase 3
高级模型：共享额度池、交叉产品额度、动态授信。

## 典型错误

1. **把额度当余额** — cash_limit_balance 这种字段是混乱设计
2. **所有子账户都建真实账户** — 门店统计也建账户，账户千万级爆炸
3. **虚拟余额允许真实出款** — 会造成资金黑洞

## 对本项目的参考价值

天枢核心系统的三条边界必须清晰：账户系统只管理真钱，虚拟子账户只管理核算维度，额度系统只管理可使用权。优先级 A > B > C。


## A 与 B 功能点对比分析

> 来源：天枢银行账务系统 spec 评审讨论（2026-04-23）

在将 A（资金隔离型 SUB_REAL）和 B（记账隔离型 VIRTUAL）落地到账务系统 spec 时，逐功能点对比发现两者在数据模型和系统行为上的差异：

| 功能点 | SUB_REAL（资金隔离型） | VIRTUAL（记账隔离型） |
|--------|----------------------|---------------------|
| 本质 | 真实资金账户，与 MAIN 同等地位 | 核算维度账本，不是资金账户 |
| 余额字段 | balance / available_balance / frozen_amount | dimension_balance（仅核算统计） |
| 余额约束 | 主账户 balance = Σ 子账户 balance | Σ dimension_balance ≤ 主账户 balance |
| 开户 | 生成独立 account_id + account_no，关联 parent_account_id | 同左 |
| 入账（IN） | 更新自身 balance + 同事务内更新主账户 balance | 双写：主账户 balance + 自身 dimension_balance |
| 出账（OUT） | 更新自身 balance（校验自身 available_balance）+ 同事务内更新主账户 balance | 禁止直接出账，需对主账户操作 |
| 借贷平衡校验 | 分录 account_id = 自身，正常参与凭证借贷平衡 | 分录 account_id = 主账户（参与借贷平衡），dimension_account_id = 自身（不参与） |
| 冲正 | 反向凭证作用于自身余额 + 同步更新主账户余额 | 反向凭证作用于主账户 balance + 自身 dimension_balance |
| 冻结/解冻 | 支持，冻结自身 frozen_amount | 不支持（无真实资金，冻结应作用于主账户） |
| 状态管理 | 完整生命周期 + 标志位（与 MAIN 一致） | 仅 lifecycle_status（ACTIVE / CLOSED），标志位无意义 |
| 销户校验 | balance=0、frozen_amount=0、标志位均 false、未结利息=0 | dimension_balance=0 即可 |
| 日终余额快照 | 快照自身 balance / available_balance / frozen_amount | 快照 dimension_balance |
| 日终发生额汇总 | 汇总自身借方/贷方发生额 | 汇总自身 dimension 借方/贷方变动额 |
| 存款计息 | 支持，按自身 balance 计息 | 不支持（无真实资金，计息在主账户） |
| 流水查询 | 按自身 account_id 查询 | 按 dimension_account_id 查询 |
| 并发控制 | 乐观锁：自身 version + 主账户 version（双锁） | 乐观锁：主账户 version |

## Phase 1 落地决策：砍掉 VIRTUAL

### 决策

Phase 1 仅实现资金隔离型子账户（SUB_REAL），不实现记账隔离型（VIRTUAL）。

### 理由

1. **数据模型高度重合**：从上表可以看出，如果将 VIRTUAL 放入 Account 表（独立 account_id、parent_account_id、生命周期状态等），其数据模型与 SUB_REAL 几乎一致，只是行为约束不同（不能出账、不能冻结、余额是统计性质）。这意味着"防止账户膨胀"这个 B 模型的核心优势不存在了——每个 VIRTUAL 子账户照样是 Account 表里的一行。

2. **增加的复杂度不值得**：VIRTUAL 引入了双写逻辑（主账户真实余额 + 子账户虚拟余额）、出金禁止校验、冲正对称双写、dimension_account_id 等额外复杂度，但 Phase 1 没有明确的"只要核算统计不要资金隔离"的业务场景来驱动。

3. **正确的 B 模型实现方式不同**：本文档"推荐数据模型"一节已经指出，B 模型应该用独立的 Ledger Dimension 表（`account_id, dimension_type, dimension_code, virtual_balance`），而不是复用 Account 模型。Phase 2 如有核算维度需求，应按此方式实现。

### 演进路径

- Phase 1：仅 SUB_REAL（资金隔离型），Account 表 account_type 枚举为 MAIN / INTERNAL / SUB_REAL
- Phase 2：如有核算维度需求，通过独立的 Ledger Dimension 表实现，不在 Account 表中增加 VIRTUAL 类型
