# 账户状态机设计决策（多维状态模型）

## 来源
- 链接：对话粘贴
- 来源：ChatGPT
- 时间：2026-04-21

## 问题

银行账户系统只设计 NORMAL / FROZEN / CLOSED 三个状态远远不够。真正银行级系统中，账户状态决定是否允许入账、出账、计息、渠道交易、司法扣划、反洗钱监控、休眠治理、销户等。如何设计完整的账户状态机？

## 结论

**账户状态不是一个字段，应拆为四个维度：**

| 维度 | 职责 | 示例值 |
|------|------|--------|
| **lifecycle_status** | 账户存在性（开户→销户） | INIT / PENDING_OPEN / ACTIVE / PENDING_CLOSE / CLOSED |
| **restriction_status** | 交易能力（冻结/止付） | NONE / PARTIAL_FREEZE / FULL_FREEZE / STOP_PAYMENT |
| **activity_status** | 运营治理（睡眠/久悬） | NORMAL / DORMANT / UNCLAIMED |
| **compliance_status** | 监管要求（黑名单/KYC） | NORMAL / KYC_LIMITED / BLACKLISTED |

一个账户可能同时处于"已开户 + 部分冻结 + 睡眠户"，单状态枚举无法表达。

## 关键论据

### 1. 生命周期状态（主状态）

| 状态 | 含义 | 说明 |
|------|------|------|
| INIT | 初始化 | 账户号已生成，未完成开户（预约/预申请/批量预开户） |
| PENDING_OPEN | 待激活 | KYC/实名/签约未完成，不可交易不可计息 |
| ACTIVE | 正常户 | 标准可用，允许入账出账冻结计息渠道交易 |
| PENDING_CLOSE | 待销户 | 销户申请处理中 |
| CLOSED | 销户 | 终态，不可交易不可恢复，可查询历史 |

迁移路径：
```text
INIT → PENDING_OPEN → ACTIVE → PENDING_CLOSE → CLOSED
```

### 2. 交易限制状态（叠加状态）

这类状态不替代 ACTIVE，而是附加在生命周期状态之上。

| 状态 | 含义 | 限制 |
|------|------|------|
| PARTIAL_FREEZE | 部分冻结 | 冻结指定金额，available = balance - frozen |
| FULL_FREEZE | 全额冻结 | 司法/风控冻结，禁止出账，通常允许入账 |
| STOP_PAYMENT | 止付 | 禁止付款/转出，但资金未冻结（账户风险调查/反诈止付/临时限制） |

**止付与冻结语义完全不同**，不可混用。

### 3. 活跃状态（运营治理）

| 状态 | 含义 | 处理 |
|------|------|------|
| DORMANT（睡眠户） | 长期无客户主动交易（如 12 个月） | 限制部分交易、加强身份校验、重新激活后恢复 |
| UNCLAIMED（久悬户） | 多年无交易且无联系 | 特殊监管报送、集中管理账户池、特殊领回流程 |

迁移路径：
```text
ACTIVE → DORMANT → ACTIVE（客户激活）
DORMANT → UNCLAIMED
```

### 4. 销户前校验规则

CLOSED 前必须校验：
- 余额 = 0
- 冻结金额 = 0
- 未结利息 = 0
- 未完成交易 = 0
- 未出账单 = 0
- 无司法限制

### 5. 数据库模型

```text
account
  account_id
  account_no
  lifecycle_status
  restriction_status
  activity_status
  compliance_status
  status_reason_code
  status_effective_time
  status_expire_time
  version
```

冻结需支持明细：
```text
freeze_start_time, freeze_end_time, freeze_reason, freeze_source
```

### 6. 事件驱动设计

状态变更应发布领域事件，供支付系统、风控系统、通知系统、对账系统订阅：

```text
AccountOpened
AccountFrozen
AccountUnfrozen
AccountDormantMarked
AccountReactivated
AccountClosed
```

### 7. 接口设计原则

状态迁移必须命令式，不要开放通用 PATCH：

```text
POST /accounts/{id}/freeze
POST /accounts/{id}/unfreeze
POST /accounts/{id}/mark-dormant
POST /accounts/{id}/reactivate
POST /accounts/{id}/close
```

**不要**：`PATCH /account status=ACTIVE`

### 8. 状态修改审计

每次状态变更必须记录：谁改的、何时改、原因、来源系统、审批单号。

### 9. 典型错误

| 错误 | 后果 |
|------|------|
| 冻结 = 销户 | 旧系统冻结后无法恢复 |
| 睡眠户只是标签 | 应实际影响交易权限 |
| 止付和冻结混用 | 语义完全不同，业务规则不同 |
| 状态修改无审计 | 无法追溯，合规风险 |

## 对本项目的参考价值

天枢核心系统的账户状态必须采用多维模型（lifecycle + restriction + activity + compliance），而非单枚举。这一设计直接影响交易准入判断、日终批量过滤、合规报送、运营治理等核心能力。Phase 1 至少落地 lifecycle_status + restriction_status，activity_status 和 compliance_status 可在 Phase 2 补充。
