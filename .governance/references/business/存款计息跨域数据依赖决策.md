# 存款计息跨域数据依赖决策（deposit-core 与 account-core 的关系）

## 来源
- 链接：对话粘贴
- 来源：ChatGPT
- 时间：2026-04-21

## 问题

deposit-core 做活期计息时需要账户状态、余额等数据，是否要实时依赖并查询 account-core？如果千万级账户日终批量计息逐笔 RPC 查询 account-core，会不会成为灾难？

## 结论

**deposit-core 需要账户数据，但批量计息时绝不应强依赖在线查询 account-core。正确方式是：账户主数据归属 account-core，计息使用本地快照。**

架构原则：**写操作看归属域，读操作看效率模型。**

## 关键论据

### 1. 计息需要的数据来自多个域

| 数据类型 | 来源 |
|---------|------|
| 账户状态、开户日、产品号 | account-core |
| 日终余额、积数、冻结额 | ledger-core |
| 利率、阶梯规则 | product-center |
| 特殊名单（VIP 等） | customer/profile |

deposit-core 需要多个域的数据，但不代表实时 RPC 查询才能运行。

### 2. 批量计息绝不依赖实时链路

千万级账户日终批量计息如果逐笔调 account-core：
- RPC 风暴
- 批量跑不完
- 链路脆弱
- 重跑困难

### 3. 正确做法：数据本地化（读模型）

```text
account-core ─────┐
                   ├──> deposit-core 本地读模型
ledger-core  ─────┤
product-center ───┘

deposit-core:
  批量计息（本地扫描）
  生成利息结果
  调 ledger-core 入账
```

同步方式：事件同步 / CDC，将账户快照和余额快照同步到 deposit-core 本地。

### 4. 两种场景区分处理

| 场景 | 方式 | 原因 |
|------|------|------|
| 开户/改单（低频实时） | 同步调用 account-core | 低频操作，可实时 RPC |
| 日终批量计息（高频海量） | 本地快照批量扫描 | 千万级不可逐笔远程查询 |

### 5. 本地快照表设计

账户快照表：
```text
deposit_account_snapshot
  account_id, customer_id, product_id, status,
  open_date, close_date, rate_plan_id,
  interest_flag, last_interest_date
```

余额快照表：
```text
deposit_balance_snapshot
  account_id, ledger_balance, available_balance,
  freeze_balance, daily_avg_balance, as_of_date
```

### 6. 为什么计息不放 account-core

account-core 是通用账户域，计息是产品逻辑。如果 account-core 自己做计息，会膨胀成：账户 + 存款 + 贷款 + 卡 + 利率 + 分期 + 营销，最终变成巨石核心。

各域职责：

| 域 | 职责 |
|----|------|
| account-core | 开户、销户、冻结、账户状态、账户归属客户、产品绑定（账户生命周期） |
| deposit-core | 活期计息、定期到期计息、利息试算、结息跑批、税费处理（存款产品域） |
| ledger-core | 借贷分录、余额更新、流水落库、总账映射（账务域） |

### 7. 数据流

```text
开户: deposit-core → account-core 开账户
计息: deposit-core 使用本地快照计算 → ledger-core 入账
```

### 8. 分阶段落地

| 阶段 | 方案 |
|------|------|
| Phase 1（简化版） | 同库不同模块，account_core.account 与 deposit_core.interest_job 通过 SQL join 计算 |
| Phase 2（演进版） | CDC 同步、独立库、事件驱动 |

## 对本项目的参考价值

天枢核心系统的 deposit-core 批量计息必须采用本地快照模式，避免对 account-core 的实时链路依赖。Phase 1 可先同库 join，但模块边界要清晰，为后续 CDC 独立库演进做准备。这一决策直接影响日终批量的可靠性和性能。
