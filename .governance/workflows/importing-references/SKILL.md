# 导入参考资料

## Overview

将用户在其他 AI（ChatGPT、Claude、DeepSeek 等）上的咨询结论、技术文章、官方文档等外部资料，提炼为结构化的参考文档，存入 `.governance/references/`。

**Announce at start:** "我正在使用 importing-references 工作流来整理参考资料。"

## 输入来源

用户可能提供以下任意形式的输入：

| 来源 | 示例 |
|------|------|
| AI 对话内容 | 直接粘贴 ChatGPT/Claude 的问答 |
| 分享链接 | https://chatgpt.com/share/xxx（可能无法访问） |
| 技术文章 | 博客、官方文档的链接或内容 |
| 会议纪要 | 粘贴的讨论记录 |
| 截图 | 包含关键信息的图片 |

## 处理流程

```
用户提供原始资料
    │
    ├─ 1. 识别主题 → 确定分类（business / technical）
    │
    ├─ 2. 提炼内容 → 问题 + 结论 + 关键论据 + 对本项目的价值
    │
    ├─ 3. 确定文件名 → 中文，简短，体现核心问题
    │
    ├─ 4. 写入文件 → .governance/references/{business|technical}/xxx.md
    │
    └─ 5. 确认 → 告知用户文件路径
```

## 文档格式

```markdown
# 问题标题

## 来源
- 链接：（原文链接，没有则写"对话粘贴"）
- 来源：（ChatGPT / Claude / 技术文章 / 会议纪要 等）
- 时间：YYYY-MM-DD

## 问题
（用一段话描述原始问题）

## 结论
（核心结论，加粗关键决策）

## 关键论据
（支撑结论的核心理由，保留有价值的细节）

## 对本项目的参考价值
（一两句话说明为什么对天枢项目有用）
```

## 提炼原则

1. **保留决策价值** — 重点是"选什么"和"为什么选"，不是过程讨论
2. **去除对话噪音** — 去掉寒暄、重复、过渡语句
3. **保留关键细节** — 数据模型、分阶段建议、典型错误等实战内容保留
4. **一个问题一个文件** — 不要把多个不相关的问题合并
5. **中文文件名** — 简短，体现核心问题，如"虚拟子账户设计决策.md"

## 分类规则

| 内容类型 | 目录 | 示例 |
|---------|------|------|
| 业务模型、业务规则、产品设计 | `references/business/` | 科目体系设计、子账户模型 |
| 技术方案、架构选型、中间件 | `references/technical/` | 分布式事务对比、缓存策略 |
| 既有业务又有技术 | 按主要决策归类 | 科目体系→business（业务边界决策） |

## 批量导入

用户可能一次性提供多段对话。处理方式：

1. 先通读全部内容，识别有几个独立问题
2. 逐个确认：列出识别到的问题清单，问用户是否遗漏
3. 逐个生成文档

## 链接无法访问时

如果用户提供的链接无法访问（需要登录等），直接告知：

> "这个链接我无法访问，请把关键内容粘贴给我，我来整理。"

不要反复尝试。
